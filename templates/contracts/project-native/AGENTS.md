# {{DISPLAY_NAME}} Repository Contract

适用范围：仓库根目录及其子目录；如果具体 workspace 或子目录存在更近的 `AGENTS.md`，则以更近者为准。

这些规则与 OMX 顶层编排契约同时生效：OMX 负责通用执行/协调机制，以下条目定义这个仓库中的项目定位、架构优先级与硬约束。

## 项目定位

- 定义这个仓库默认代表什么，而不是什么
- 定义人类、Agent 和平台各自的职责边界
- 定义优先通过哪些稳定契约推进任务，避免仓库退化成临时脚本集合

## 设计与实现优先级

- 定义本仓库的顶层入口与首选抽象链路
- 定义哪些能力应通过 profile / overlay / controller / adapter 等稳定层表达
- 定义 README 面向谁、底层接口说明应该落在哪里
- 定义当前主线优先级，避免被上游或旁支工作无节制稀释
- 定义第三方 Agent 接入时必须遵循的稳定接口面

## 主线稳定性与 Worktree 规则

- 定义共享 checkout 是否应被视为稳定依赖面
- 定义 `main` 的稳定性要求
- 定义什么情况下必须使用独立 worktree
- 定义项目内部推荐的 worktree 管理目录

## 文档分层

- 定义 `guides/`、`docs/` 等目录分别承载什么内容
- 定义哪些文档属于公开稳定面，哪些属于内部过程产物

## 数据与状态变更

- 读状态优先于做变更
- 数据资产或状态变更优先走统一 controller / mutation 入口
- 重要 mutation 必须留下可审计落盘结果
- 非法 payload / 契约 / 状态应显式报错，不要静默纠偏

## 研究 / 业务推进偏置

- 在这里定义本项目优先鼓励的推进方向
- 说明哪些弱结果、低价值方向应优先止损而不是继续空转

## 人类审核面

- 定义人类主要审核哪些正式落盘结果
- 定义 Agent 应把哪些关键判断写入可审计表面

## 冲突处理

- 用户显式指令高于本文件
- 更近目录下的 `AGENTS.md` 高于本文件
- 当 OMX 通用执行规则与仓库特定规则同时适用时：前者负责如何编排执行，后者负责本仓库内什么目标、边界与约束优先
